# Workflow: 核心邏輯與架構開發 (/plan-core)

**描述:** 用於引導 Agent 從零開始釐清全新功能的底層邏輯、資料結構與外部依賴，並負責完成核心程式碼的實作與自我審查。（不包含具體的 UI 視圖渲染）

---

## 步驟 1：核心需求與資料邊界澄清 (Clarification)
**[Agent 執行指示]**
1. 閱讀使用者的初始需求。**嚴禁在此階段生成任何實作程式碼。**
2. 針對資料模型、狀態同步、網路處理或硬體限制提出釐清問題。
3. **動態提問原則：** 不限制問題的數量，請根據需求的複雜度提出所有必要的疑問。
4. **選項強制原則：** 對於提出的每一個問題，**必須同時提供多個專業的實作建議或選項**，並附上各自的優缺點，讓使用者能直接挑選或作為參考。
5. **暫停執行**，等待使用者回覆選擇。

## 步驟 2：邏輯架構草案設計 (Architecture Draft)
**[Agent 執行指示]**
1. 根據使用者的選擇，產出核心邏輯的技術規格草案。
2. 內容需包含：
   - 資料模型 (Models) 的具體欄位與結構定義。
   - 負責處理業務邏輯的 Manager / Service 類別介面 (Interfaces/Protocols)。
   - 異常處理與重試機制 (Error Handling & Retries)。
3. 詢問使用者：「請問這份架構草案是否符合預期？輸入『確認』進入實作階段，或提出修改建議。」
4. **暫停執行**，等待使用者確認。

## 步驟 3：分層實作程式碼 (Implementation)
**[Agent 執行指示]**
1. 獲得使用者確認後，嚴格依照步驟 2 的草案開始撰寫程式碼。
2. **防截斷機制：** 如果程式碼量較大，請將實作拆分為合理的區塊逐步提供（例如：先提供資料層程式碼，再提供網路請求層，最後提供業務邏輯層），避免一次輸出過多。
3. 確保程式碼包含必要的註解，特別是在處理複雜非同步邏輯或極端情況的地方。

## 步驟 4：自我審查與測試建議 (Review & Testing)
**[Agent 執行指示]**
1. 程式碼生成完畢後，執行自我審查：
   - 是否遺漏了步驟 1 中討論到的極端情況？
   - 變數命名與架構是否符合專案慣例？
2. 提供一段具體的「測試建議」或「單元測試範例」，告訴使用者該如何驗證這段剛寫好的核心邏輯是否運作正常。
3. 流程結束。
